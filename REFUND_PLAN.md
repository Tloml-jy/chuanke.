# 退款闭环功能开发计划

## 一、现状分析

### 1. 数据模型（orders集合）
**已有字段：**
- `paymentStatus`: paid/partial_refunded/refund_processing/refunded
- `refundApplyStatus`: pending_review/approved/rejected/processing/partial_success/success/failed
- `refundReason`: 退款原因
- `refundRequestedAmount`: 申请退款金额（分）
- `refundAmount`: 实际退款金额（分）
- `refundDeductionFee`: 扣除费用（分）
- `refundDeductionRate`: 扣除费率
- `refundApplyTime`: 申请时间
- `refundReviewTime`: 审核时间
- `refundReviewBy`: 审核人
- `partialRefunds`: 部分退款记录数组
- `operationLogs`: 操作日志数组

### 2. 云函数接口（order/index.js）
**已有接口：**
- `refund`: 商户发起退款（兼容入口，直接审核通过+执行）
- `cancelOrder`: 用户取消订单
- `getMerchantOrderDetail`: 商户订单详情
- `getOrderDetail`: 用户订单详情

**缺失接口：**
- ❌ `applyRefund`: 用户申请退款
-  `reviewRefund`: 商户审核退款（通过/驳回）
- ❌ `executeRefund`: 执行退款（调用微信退款API）

### 3. 前端页面
**用户端（order/detail）：**
- ✅ 显示退款状态
- ✅ 显示可退金额
- ✅ 有"申请退款"按钮（但功能不完整）

**商户端（merchant/order-detail）：**
- ✅ 显示退款状态
- ✅ 显示退款金额信息
- ✅ 有退款相关按钮（但逻辑不完善）

**缓存刷新机制：**
- 使用 `markDirty()` 标记缓存失效
- 用户端：`orders:mixed`
- 商户端：`merchant:order:list`

## 三、状态机设计

```
待支付(unpaid)
    ↓ 支付成功
已支付(paid)
    ↓ 用户申请退款
退款待审核(refund_pending_review)
    ↓ 商户审核通过
退款已审核(refund_approved)
    ↓ 执行退款
退款处理中(refund_processing)
    ↓ 微信退款成功
退款成功(refunded)

或者：
    ↓ 商户审核驳回
退款已驳回(refund_rejected) → 回到已支付状态
```

**部分退款状态：**
```
已支付(paid) → 部分退款成功(partial_refunded) → 再次申请退款 → ...
```

## 六、风险评估

### 高风险
1. **并发退款**：用户快速点击多次申请退款
   - 解决：数据库唯一索引 + 幂等性检查

2. **退款金额超付**：部分退款累计超过订单总额
   - 解决：每次退款前计算可退金额

3. **微信退款API失败**：网络异常或商户号余额不足
   - 解决：支持重试机制，记录失败原因

### 中风险
1. **状态不一致**：订单状态和退款记录不同步
   - 解决：使用事务保证原子性

2. **缓存未刷新**：退款后页面未更新
   - 解决：markDirty机制 + 手动刷新按钮
