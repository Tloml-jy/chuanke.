## UI 组件库与选型（AI 速查）

> 目的：后续开发时优先查本节，再决定用原生还是第三方组件；新页面按需 `usingComponents` 引入，避免全量注册拖慢包体。

### 1. 为什么选 Vant Weapp

- 与 **微信小程序原生** 兼容好，文档成熟，社区用例多。
- 覆盖本项目高频场景：**搜索、表单、弹窗、空态、加载、标签、步骤条、列表单元格** 等。
- 可渐进接入：某一页先用 `van-field` + `van-button`，不必一次改全站。

### 2. 仓库内已做的配置


| 项                          | 说明                                                                       |
| -------------------------- | ------------------------------------------------------------------------ |
| `miniprogram/package.json` | 依赖 `@vant/weapp`                                                         |
| `project.config.json`      | `packNpmRelationList` 指向 `miniprogram/package.json`，构建输出到 `miniprogram/` |
| 本地安装                       | 在 `miniprogram` 目录执行 `npm install`（已可执行）                                 |


**开发者工具必做一步**：菜单 **工具 → 构建 npm**，生成 `miniprogram/miniprogram_npm/`。  
构建成功后，页面或 `app.json` 中通过 `**@vant/weapp/组件名/index`** 引用（见下）。

### 3. 引用方式（示例）

**全局注册（适合多处复用的少数基础组件）** — 在 `miniprogram/app.json` 的 `usingComponents` 增加：

```json
"van-button": "@vant/weapp/button/index"
```

**页面级注册（推荐，便于控制包体）** — 在页面同名 `*.json`：

```json
{
  "usingComponents": {
    "van-field": "@vant/weapp/field/index",
    "van-cell-group": "@vant/weapp/cell-group/index"
  }
}
```

WXML：`<van-button type="primary" bind:click="onTap">确定</van-button>`

### 4. 与本项目匹配的 Vant 组件清单（按场景）


| 业务场景                      | 推荐组件                              | 说明                                          |
| ------------------------- | --------------------------------- | ------------------------------------------- |
| 商品/服务/订单/预约 **搜索框**       | `Search`、`Field`                  | 统一搜索条、带清空；复杂筛选可配 `DropdownMenu`             |
| **表单**（地址、核销码、客服配置、公告编辑）  | `Field`、`Cell`、`CellGroup`、`Form` | 标签、校验、禁用态与项目内「防连点」配合使用                      |
| **弹层/抽屉**（公告详情、拒绝原因、二次确认） | `Popup`、`Dialog`、`ActionSheet`    | 替代部分自定义 mask；长文用 `Popup` + 内嵌 `scroll-view` |
| **按钮与操作区**                | `Button`、`SubmitBar`              | 与现有 `btn-primary` 并存时可逐步统一主色                |
| **加载与禁用反馈**               | `Loading`、`Skeleton`              | 列表首屏、详情骨架；注意与 `request.js` 的 loading 不要重复遮罩 |
| **空状态**                   | `Empty`                           | 可与现有 `empty-state` 二选一或并存                   |
| **标签/状态**                 | `Tag`、`NoticeBar`                 | 订单状态、公告紧急程度；注意与现有 badge 样式统一                |
| **列表行**                   | `Cell`、`SwipeCell`                | 设置类、我的订单行；滑动删除需谨慎与滚动冲突                      |
| **分页与数量**                 | `Stepper`                         | 购物车数量（若后续扩展）                                |
| **图片展示**                  | `Image`、`Uploader`                | 商品图、商户资质（若后续接入上传）                           |
| **轻提示**                   | `Toast`                           | 可选；项目内已大量 `wx.showToast`，迁移非必须              |


**建议渐进策略**：新功能优先 Vant；旧页保持稳定，**不因接入组件库大范围改版**。

### 7. AI 开发时的使用约定

1. 新增页面若用到 Vant，在 **页面 json** 声明 `usingComponents`，路径一律 `**@vant/weapp/.../index`**。
2. 改样式时优先用 Vant 的 **外部样式类**（如 `custom-class`）或 CSS 变量，避免 `!important` 覆盖组件内部。
3. 与现有工具的配合：**云函数仍走 `utils/request.js`**；Vant 只解决展示层。
4. 提交前确认本地已 **构建 npm**，且真机/模拟器能打开用到新组件的页面。

## 📋 开发检查清单

### 前端开发

- 页面文件齐全（js/json/wxml/wxss）
- 使用封装的 `call` 函数调用云函数
- 所有异步操作有错误处理
- 列表使用 wx:key
- 支持下拉刷新和上拉加载
- 空状态有友好提示
- 图片使用懒加载
- 避免频繁 setData

### 云函数开发

- 入口有 action 路由映射
- 校验登录态和角色权限
- 关键操作使用 runTransaction
- 统一返回格式 `{ code, message, data }`
- 有结构化日志（console.log/error）
- 分页查询支持 page/pageSize
- 有完善的错误处理

### 数据库设计

- 字段类型合理（价格用 number 单位分）
- 必要字段建立索引
- 适度冗余避免联表查询
- 时间字段用 date 类型
- 状态字段用 string 枚举

## 🔍 常见问题 FAQ

### Q1: 如何查找某个功能的实现？

**A**: 

1. 确定功能所属模块（商品/订单/预约等）
2. 在 `cloud-functions.md` 中查找对应云函数和 action
3. 在 `business-flows.md` 中查看业务流程
4. 参考 `development-guide.md` 的代码模板

### Q2: 价格为什么用"分"做单位？

**A**: 

- JavaScript 浮点数精度问题（0.1 + 0.2 ≠ 0.3）
- 用整数（分）避免精度丢失
- 前端展示时除以 100 转换为元
- 工具函数：`fenToYuan(fen)` / `yuanToFen(yuan)`

### Q3: 如何处理并发冲突（如同一时段被多人预约）？

**A**: 

1. 数据库层面： bookingDate + bookingSlot + serviceId 建立唯一索引
2. 云函数层面：创建预约前再次校验时段可用性
3. 前端层面：提交时显示 loading，防止重复点击
4. 失败处理：提示"时段已被预约"，引导重新选择

### Q4: 云函数调用超时怎么办？

**A**: 

1. 优化数据库查询（加索引、减少查询次数）
2. 拆分复杂逻辑到多个云函数
3. 设置云函数超时时间（最长 60 秒）
4. 异步处理非关键逻辑（如发送订阅消息）

### Q5: 如何实现角色切换（用户 ↔ 商户）？

**A**: 

1. 调用 `user/setRole` 云函数切换角色
2. 更新 app.globalData.userInfo.role
3. 自定义 TabBar 根据角色显示不同 tab
4. 路由守卫检查页面访问权限

### 云数据库与部署说明（本次新增）

- ✅ 本次依赖集合：`system_settings`（需先存在）
- ✅ 你当前采用“手动建集合”方式，推荐步骤：
  1. 在云开发控制台手动创建集合：`system_settings`
  2. 重新部署云函数：`system`、`bootstrap`
  3. 调用 `bootstrap`：`{ "action": "initHomeNoticeConfig" }`
  4. 如需初始化某个商户公告模板，调用 `bootstrap`：`{ "action": "initMerchantNoticeConfig", "merchantId": "merchant_demo_001" }`
  5. 在 `system_settings` 中确认存在文档：`_id = home_notice`、`_id = merchant_notice_<merchantId>`
- ✅ `home_notice` 文档示例：
  - `_id`: `home_notice`
  - `enabled`: `true / false`
  - `notices`: 公告数组（推荐）
    - `id`: 公告ID
    - `text`: 公告内容
    - `priority`: 优先级（数字越大越靠前）
    - `startTime/endTime`: 生效时间范围（可选）
    - `enabled`: 单条开关
    - `order`: 同优先级时排序字段
  - `noticeText`: 兼容旧版单条文本字段
- ✅ `merchant_notice_<merchantId>` 文档示例（商户公告，只在商户工作台展示）：
  - `_id`: `merchant_notice_merchant_demo_001`（示例）
  - `enabled`: `true / false`
  - `notices`: 公告数组（字段与 `home_notice.notices` 一致）
  - `noticeText`: 兼容旧版单条文本字段
- ✅ 管理员维护SOP（建议）
  1. 先在 `system_settings` 中编辑目标文档（平台：`home_notice`；商户：`merchant_notice_<merchantId>`）
  2. 单条公告至少填写：`id`、`text`、`enabled`
  3. 建议同时填写：`tag`（紧急/活动/通知）、`priority`、`publishTime`
  4. 如需定时生效，填写 `startTime/endTime`（格式建议：`YYYY-MM-DD HH:mm:ss`）
  5. 修改后下拉刷新首页/商户工作台，验证展示顺序与时间过滤是否符合预期
  6. 若目标商户尚无公告文档，可先调用 `initMerchantNoticeConfig` 生成模板后再编辑
- ✅ 排障建议（管理员）
  - 页面无公告：先检查文档 `enabled` 和单条 `enabled` 是否都为 `true`
  - 公告不在列表：检查是否命中过滤（未到 `startTime` 或超过 `endTime`）
  - 排序不符合预期：检查 `priority`（高优先）与 `order`（同优先级升序）
  - 标签不正确：优先填写显式 `tag`，避免仅依赖关键词识别
